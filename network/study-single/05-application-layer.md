## 응용 계층

### DNS
- 도메인 네임(Domain Name) : IP 주소와 대응되는 문자열 형태의 호스트 정보, 네임 서버가 관리 (ex. `www.google.com`)
  - '.'을 기준으로 계층적으로 분류되며, 최상단에 루트 도메인(`.`), 다음 단계로 최상위 도메인(`com`), 2단계 도메인(`google`), ...이 있다.
- 호스트 네임 : 전체 도메인 네임의 첫 번째 부분, 호스트를 식별할 수 있다는 의미 (`www`)

- DNS(Domain Name System) : 계층적이고 분산된 도메인 네임에 대한 관리 체계이자 이를 관리하는 프로토콜 (계층적 구성, 반복적 질의 진행)
  - 로컬 네임 서버(로컬 DNS 서버, 리졸버) : 클라이언트와 맞닿아 있는 네임 서버, 일반적으로 ISP에서 할당해 줌
  - 루트 네임 서버(루트 DNS 서버) : 루트 도메인을 관장하는 네임 서버, TLD 네임 서버의 IP 주소를 반환함
  - TLD 네임 서버(TLD 서버) : TLD를 관리하는 네임 서버, TLD의 하위 도메인 네임을 관리하는 네임 서버 주소를 반환함
  - 책임 네임 서버(책임 서버) : 특정 도메인 네임을 관리하는 서버, 자신이 관리하는 도메인 영역의 질의에 대해 곧바로 대답할 수 있는 서버
- DNS 캐시 : TTL 동안의 시간만큼 도메인 네임의 질의 결과를 임시 저장하는 것을 의미

- DNS 레코드 타입
  - A : 특정 호스트에 대한 도메인 네임과 IPv4와의 대응 관계
  - AAAA : 특정 호스트에 대한 도메인 네임과 IPv6와의 대응 관계
  - CNAME : 호스트 네임에 대한 별칭 지정
  - NS : 특정 호스트의 IP주소를 찾을 수 있는 네임 서버
  - MX : 해당 도메인과 연동되어 있는 메일 서버

### 자원과 URI
- 자원(resource) : 네트워크상의 메시지를 통해 주고받을 수 있는 대상
- URI(Uniform Resource Identifier) : 자원을 식별할 수 있는 정보
  - URL(Uniform Resocuce Locator) : 위치를 이용해 자원을 식별
  - URN(Uniform Resource Name) : 이름을 이용해 자원을 식별 (ex. `urn:isbn:0451450523`)
- URL의 구성 (ex. `https://www.google.com:443/search?q=google#image`)
  - scheme(스키마) : 자원에 접근하는 방식 (`https`)
  - authority : 호스트 특정 정보, IP 주소 or 도메인 네임 + 포트로 구성 (`www.google.com:443`)
  - path : 자원이 위치한 경로 (`/search`)
  - query : 관련 추가 정보, key-value 형식으로 구성 (`?q=google`)
  - fragment : 자원의 한 조각을 가리키기 위한 정보

### HTTP
- HTTP(Hypertext Transfer Protocol) : 응용 계층에서 정보를 주고받는데 사용하는 프로토콜
  - **요청-응답 기반 프로토콜** : 클라이언트-서버 구조 기반이며 요청/응답 메시지 형식이 다름
  - **미디어 독립적 프로토콜** : 주고받을 자원의 특성과 무관하게 그저 자원을 주고받을 수단(인터페이스)의 역할만을 수행함
  - **스테이트리스 프로토콜** : 서버가 클라이언트와 관련된 상태를 기억하지 않음
  - 지속 연결 프로토콜 : 최근 대중적으로 사용하는 HTTP 버전(HTTP 1.1 이상)은 하나의 TCP 연결에서 여러 개의 요청/응답을 주고받을 수 있다.

- HTTP의 발전
  - HTTP/0.9 : GET 메서드만 사용 가능, 요청 메시지는 한 줄로 구성
  - HTTP/1.0 : 이외 메서드 도입, 헤더 지원, 지속 연결 지원 X
  - HTTP/1.1
    - 지속 연결을 공식 지원
    - 파이프라이닝 기능(응답 수신 전 다른 요청 가능) 추가
    - 다양한 편의 기능 및 헤더 추가
  - HTTP/2.0
    - 효율과 성능 개선 버전 (헤더 압축 전송, 바이너리 데이터 기반 송수신, 서버 푸시 기능 제공)
    - 멀티플렉싱 기법을 도입하여 HOL 블로킹(Head-Of-Line Blocking) 해결
  - HTTP/3.0
    - UDP를 기반으로 구현된 QUIC(Quick UDP Internet Connections) 프로토콜 기반
    - UDP는 상대적으로 더 빠르기 때문에 속도면에서 큰 개선이 이루어짐
