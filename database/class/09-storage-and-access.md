## DB의 저장과 접근

1. 디스크 저장과 접근의 기본
  - 목표: 데이터베이스 성능 향상의 핵심은 가장 느린 작업인 디스크 접근 횟수(I/O)를 최소화하는 것
  - 저장 레코드 식별
    - 각 레코드는 디스크 내에서 유일한 RID(Record Identifier)로 식별되며, 이는 `<페이지 번호, 오프셋(슬롯 번호)>`으로 구성됨
    - 레코드의 위치가 페이지 내에서 변경되어도 오프셋 내용만 변경하면 되므로 RID 자체는 변하지 않는다
2. 파일 조직 방법 : 데이터를 디스크에 배치하는 방식입니다.
  - 엔트리 순차 파일(Heap/Pile): 데이터가 시스템에 삽입되는 순서대로 저장
  - 키 순차 파일: 레코드들의 키 값 크기 순으로 물리적으로 정렬되어 저장

3. 인덱스(Index) : 데이터를 빠르게 찾기 위해 `<탐색 키 값, 레코드 주소>` 쌍을 저장하는 구조입니다.
  - 집중(Clustering) vs 비집중(Non-clustering): 탐색 키 값의 순서와 실제 데이터 레코드의 물리적 저장 순서가 동일하면 집중 인덱스(파일당 단 1개만 가능), 무관하면 비집중 인덱스
  - 밀집(Dense) vs 희소(Sparse): 모든 탐색 키에 대해 인덱스 엔트리가 존재하면 밀집 인덱스, 일부 키에 대해서만 존재하면 희소 인덱스
  - B-트리와 B+-트리
      - **B-트리**: 디스크 접근 횟수를 줄이기 위해 트리의 높이를 최소화한 균형 잡힌 다진 검색 트리입니다.
      - **B+-트리**: 실제 RDBMS에서 가장 많이 쓰이는 구조. 데이터를 찾는 경로만 제공하는 '인덱스 세트(내부 노드)'와 실제 데이터 주소를 순차적인 연결 리스트 형태로 묶어둔 '순차 세트(리프 노드)'로 나뉩니다. 이 연결 리스트 덕분에 범위(Range) 및 순차 검색에 매우 탁월합니다.

4. **해싱(Hashing)** : 키 값을 해시 함수에 넣어 목표 데이터가 있는 물리적 주소(버킷)로 직접 접근하는 방식
  - **충돌(Collision)** 해결: 다른 키가 같은 버킷 주소로 변환되는 현상입니다. 같은 주소의 데이터들을 연결 리스트로 묶는 체이닝(Chaining) 방법과, 빈자리를 찾아 저장하는 개방 주소 방법(선형 조사, 이차원 조사 등)으로 해결
  - 확장성 해싱(Extendible Hashing): 충돌이 많아질 경우 모조 키(pseudokey)의 비트열과 디렉터리 구조를 활용해 버킷과 디렉터리 크기를 동적으로 확장해 나가는 기법

5. 비트맵 인덱스(Bitmap Index)
  - 성별이나 소득 수준(구간)처럼 가능한 값의 종류가 적을 때 유리한 인덱스
  - 데이터를 0과 1의 비트 배열로 저장하므로, 다중 조건 질의 시 교집합(AND)이나 합집합(OR) 같은 비트 연산을 통해 매우 빠르게 결과를 도출할 수 있다.
