## 병행 제어(Concurrency Control)

1. 병행 제어의 필요성
  - 갱신 분실(Lost Update): 두 트랜잭션이 동시에 같은 데이터를 덮어써서 한쪽의 갱신 내용이 사라지는 현상
  - 모순성(Inconsistency): 트랜잭션이 실행되는 도중에 다른 트랜잭션이 개입하여 데이터베이스의 일관성이 깨지는 현상
  - 연쇄 복귀(Cascading Rollback): 한 트랜잭션이 실패하여 취소(Rollback)될 때, 그 트랜잭션이 변경한 데이터를 읽어간 다른 트랜잭션들도 연쇄적으로 취소되어야 하는 현상

2. 직렬 가능성(Serializability) : 여러 트랜잭션이 섞여서 실행(비직렬 스케줄)되더라도, 그 결과가 트랜잭션들을 순서대로 하나씩 실행(직렬 스케줄)한 것과 똑같아야 한다
  - 충돌 직렬 가능(Conflict Serializable): 충돌이 일어나는 연산(최소 하나가 기록(write) 연산인 경우)의 순서가 직렬 스케줄과 동일한 상태
  - 선행 그래프(Precedence Graph): 트랜잭션 간의 충돌 관계를 방향 그래프로 그렸을 때, 사이클(Cycle)이 없다면 충돌 직렬 가능함을 의미

3. **Lock** : 데이터에 접근하기 전 잠금(Lock)을 걸어 다른 트랜잭션의 간섭을 막는 기법
  - 락의 종류: 읽기 연산만 허용하는 **공용 락(S-lock)**와, 읽기와 쓰기를 모두 허용하는 **배타 락(X-lock)**가 있습니다.
  - 2단계 로킹 규약(2PLP, Two-Phase Locking Protocol): 트랜잭션이 Lock만 수행할 수 있는 '확장 단계(Growing phase)'와 Unlock만 수행할 수 있는 '축소 단계(Shrinking phase)'로 나누어 실행하는 규칙 (직렬 가능성 보장)
  - 엄밀 2단계 로킹(Strict 2PLP): 연쇄 복귀를 막기 위해 트랜잭션이 완료(Commit)될 때까지 베타 락(X-lock)를 해제하지 않는 방식 (대부분의 상용 DBMS가 사용)
- **비관적 락 vs 낙관적 락**
  - **비관적 락(Pessimistic Lock)**: 실제로 데이터에 **락을 걸어서 다른 트랜잭션을 차단**하는 방식입니다. (ex. S-Lock, X-Lock)
  - **낙관적 락(Optimistic Lock)**: 실제로 락을 걸지 않고, **데이터의 버전 정보를 비교**하여 수정 시점에 충돌 여부를 확인하는 방식
- **교착 상태(Deadlock)**: 두 트랜잭션이 서로가 가진 데이터의 락가 풀리기를 무한정 기다리는 상태

4. 타임스탬프 순서(Timestamp Ordering) 기법
  - 트랜잭션이 시스템에 들어온 시간(클록 값 등)을 기준으로 고유한 타임스탬프를 부여하고, 이 순서대로 데이터에 접근하도록 강제하는 방법
  - 락를 사용하지 않으므로 교착 상태(Deadlock)가 발생하지 않지만, 연쇄 복귀가 일어날 수 있다는 단점이 있다
  - Thomas의 기록 규칙: 더 최근의 트랜잭션이 이미 데이터를 덮어썼다면, 과거 트랜잭션의 무용지물인 기록(Write) 연산을 무시해버림으로써 병행성을 더욱 높이는 기법
