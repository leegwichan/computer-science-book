## 트랜잭션과 회복

-  **트랜잭션(Transaction)** : **데이터베이스 상태를 변화시키는 작업의 논리적 단위**
- **트랩잭션의 특징(ACID)**
  - **원자성(Atomicity)**: 트랜잭션의 연산은 데이터베이스에 '모두 반영되든지(All) 아니면 전혀 반영되지 않아야(Nothing)' 한다.
  - **일관성(Consistency)**: 트랜잭션 실행이 성공적으로 완료되면 언제나 일관성 있는 데이터베이스 상태로 변환되어야 한다.
  - **격리성(Isolation)**: 트랜잭션 실행 중 연산의 중간 결과에 다른 트랜잭션이 접근할 수 없다.
  - **영속성(Durability)**: 성공적으로 완료(Commit)된 트랜잭션의 결과는 시스템이 고장 나더라도 영구적으로 반영되어야 합니다(안정 저장장치 활용).

2. **완료(Commit)와 복귀(Rollback)**
  - Commit: 트랜잭션이 성공적으로 실행되어 갱신된 데이터가 영구적으로 보장되는 일관된 상태입니다.
  - Rollback: 트랜잭션 실행이 실패하여, 수행한 모든 연산 결과를 취소(UNDO)하고 모순된 상태를 이전으로 되돌립니다.

3. 장애와 회복(Recovery)의 기초
  - 회복: 시스템 장애 발생 시, 데이터베이스를 장애 발생 이전의 오류 없는 '일관된 상태(Consistent state)'로 복원시키는 것입니다.
  - 회복의 원리: 데이터를 중복 저장하는 덤프(Dump)와 변경 내역을 기록하는 로그(Log)를 활용합니다.
  - 회복 조치(면접 빈출):
      - REDO(재실행): 장애 발생 전 완료된 기록을 바탕으로 변경된 내용(새로운 값)을 다시 실행하여 복원
      - UNDO(취소): 실행 중이었으나 완료되지 않은 변경 사항을 로그에 기록된 예전 값으로 되돌려 취소합니다.

4. 로그 기반 회복 기법의 종류 
  - **지연 갱신(Deferred Update)**: 트랜잭션이 '부분 완료'될 때까지 데이터베이스에 대한 물리적인 쓰기 연산을 지연시키는 방법. 디스크에는 쓰지 않고 로그만 남기기 때문에 장애 시 UNDO가 필요 없고 REDO만 수행하면 된다.
  - **즉시 갱신(Immediate Update)**: 트랜잭션 실행 중 데이터 변경 결과를 데이터베이스에 바로 반영하는 방식. 장애 발생 시 완료되지 않은 갱신을 취소해야 하므로 UNDO와 REDO 모두 필요하다.

5. 검사 시점 회복(Checkpoint Recovery)
  - 장애 발생 시 로그 전체를 처음부터 조사하여 REDO/UNDO를 결정하면 시간이 너무 오래 걸립니다. 이를 해결하기 위해 일정 간격으로 메모리의 버퍼 블록을 디스크에 저장하고 로그에 체크포인트를 남김.
  - 장애 발생 시, 검사 시점 이후의 로그만 탐색하여 작업 목록을 나눔
  - 완료되지 않은 트랜잭션은 Undo-list에 넣어 역순으로 취소(후진 회복)하고, 완료된 트랜잭션은 Redo-list에 넣어 순서대로 재실행(전진 회복)합니다.

6. 그림자 페이징 기법(Shadow Paging) : 로그를 사용하지 않고, 현재 페이지 테이블과 그림자 페이지 테이블 2개를 유지하는 기법입니다.
  - 실행 중에는 현재 페이지만 변경하며, 장애 발생 시 간단히 현재 페이지를 버리고 보관해둔 그림자 페이지를 복구하면 되므로 UNDO/REDO가 필요 없다.
  - 하지만 트랜잭션 완료 시 오버헤드가 발생하고, 데이터 단편화 및 병행 제어가 어렵다는 단점이 있다.
